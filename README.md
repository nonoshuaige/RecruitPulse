## 1. 简历导入 — 确定性落库

### 1.1 核心设计原则

**人数完整性由 HR 前置录入保证，数据丰富度由 AI 聊天分析补全，数据绝对准确性由求职者 / HR 最终确认闭环。**

### 1.2 底层数据结构

系统以 **姓名 + 手机号** 为复合主键，学历、工作、项目等动态字段采用一对多数组存储。

| 模块 | 字段 | 类型 | 说明 |
|------|------|------|------|
| 基础主键 | name / phone | String | 复合唯一标识，禁止 AI 修改 |
| 静态信息 | gender / age / email | String / Int / String | AI 提取，可空，留待核对 |
| 学历（动态数组） | education_history | Array[Object] | school、major、period |
| 工作（动态数组） | work_history | Array[Object] | company、role、period、content |
| 项目（动态数组） | project_history | Array[Object] | project_name、period、project_role、project_content |
| 底层文件 | resume_file | File/Link | 原始简历挂载在主键下，用于最终人工核验 |

### 1.3 四步导入流程

**步骤 1：绝对录入**
HR 输入候选人姓名和手机号，上传附件简历。系统立即生成主键记录，人数统计在此时 100% 锁定——AI 的任何波动绝不影响该记录的存在。

**步骤 2：素材归集**
HR 将与求职者的企微聊天记录、面试评语或求职者文字自述，一键粘贴到该候选人主键下的"原始素材沟通框"中。

**步骤 3：AI 结构化抽提**
系统将附件简历文本 + 原始素材沟通框文本合并，调用大模型进行针对性解析。AI 严格按照数据库字段要求输出结构化 JSON，彻底避免多候选人数据串行。

**步骤 4：人机协同确认（双通道）**
系统将 AI 提取的完整信息生成草稿表单，提供两种确认闭环方式：

- **通道 A（HR 视图）**：HR 预览 AI 填好的学历、工作、项目等动态表格，确认无误一键入库。
- **通道 B（求职者自审 - 推荐）**：系统向求职者发送链接，展示 AI 预填的完整简历。求职者自行微调或补全后提交，将大模型 95% 准确率通过人工校验提升至 100%。

### 1.4 当前前端实现

- **新增候选人**：姓名 + 手机号录入，自动校验格式与去重
- **导入聊天记录**：支持标准 JSON 文件（含 role/content/type/time 字段），FileReader 读取
- **AI 智能解析**：基于正则 + 关键词匹配，从聊天记录中提取教育/工作/项目信息（后续接入真实大模型 API）
- **结构化编辑**：动态表单支持增删学历、工作、项目条目
- **推送确认**：将解析结果推送给候选人复核

## 2. 简历筛选 — AI 动态打分与兜底

### 2.1 面试官输入端

面试官可以用自然语言输入期望特征，无需学习复杂筛选语法：

> *"我们团队缺一个能立刻上手的 Java 后端。硬性指标：本科以上。期望特质：有电商或高并发项目经验，做过核心重构优先，最好在上一家公司待满 2 年以上，稳定性好。"*

### 2.2 AI 三维评估模型

系统将面试官需求与候选人结构化数据打包投递给大模型，AI 针对三个维度客观打分：

| 维度 | 权重 | 说明 |
|------|------|------|
| 硬性匹配 | 35% | 学历、岗位对口度等硬指标 |
| 经验匹配 | 45% | 项目经验与需求关键词的契合度（高并发、重构经历等） |
| 稳定性 | 20% | 跳槽频次、就职时长评估 |

AI 对每位候选人返回：综合得分、各维度分数、推荐理由、风险提示。

### 2.3 面试官协同看板

- **按分数降序排列**：候选人池自动按综合得分从高到低呈现
- **一目了然的画像**：列表展示姓名、AI 综合得分、核心推荐理由（一句话摘要）、附件简历
- **快捷决策流**：每行右侧两个核心按钮 —— **发起面试**（进入下一阶段）或 **不合适/淘汰**（触发感谢信）

### 2.4 7 天超时兜底机制

为防止优秀人才在池中被遗忘，系统引入基于时间触发器的兜底机制：

**触发条件：** 候选人状态为"待筛选"且最后状态变更时间超过 7 天。

**动作 A — 企微催办提醒（每日 9:30）：**
> ⏰ 有 3 位候选人超过 7 天未被筛选：[张三 - Java开发] 7天未响应 → [一键转推荐] | [发起淘汰感谢信]

**动作 B — AI 跨部门推荐：**
HR 点击"转推荐"时，AI 根据候选人简历特性，自动匹配公司内其他招聘部门。若全公司无合适岗位，发送措辞温暖、尊重的拒绝信，正式闭环。

### 2.5 当前前端实现

- 自然语言输入筛选条件 → 模拟打分排序
- 候选人按综合评分降序排列
- 点击卡片查看结构化详情与原始聊天记录
- **通过** → 进入面试流程池 / **淘汰** → 填写理由归档

## 3. 面试流程 — 多轮流转与闭环

### 3.1 底层流转表

为支撑多轮面试的动态追加与历史溯源，底层以候选人主键关联面试记录子表：

| 字段 | 说明 | 约束 |
|------|------|------|
| round_number | 第几轮 | 1=初试, 2=复试, 3=终审 |
| interviewer_id | 面试官标识 | 用于精准推送消息 |
| available_slots | 面试官空闲时段 | 如 ["6/26 14:00-15:00", ...] |
| selected_slot | 候选人选定时间 | 拒绝则为空 |
| status | 单轮状态 | 0:待选择 → 1:已预约 → 2:已拒绝 → 3:待评价 → 4:通过 → 5:淘汰 |
| evaluation | 面试官评语 | 面试结束后填写 |

### 3.2 四步流转闭环

```
面试官提期 → 候选人选期 → 超时催评 → 评价晋级
```

**第一步：面试官一键"提期"**
面试官通过筛选后，在日历看板上勾选 3-5 个空闲时间段。系统锁定这些时段，生成带时效性的专属 H5 链接，自动通过企微发送给候选人。

**第二步：候选人自主预约**
候选人打开 H5 页面选择时间。接受 → 状态变为"已预约"，自动同步面试官日历。拒绝 → 状态变为"已拒绝"，HR 看板高亮红灯。

**第三步：面试超时触发器**
面试结束 1 小时后仍未评价，企微机器人向面试官推送不可屏蔽的强交互卡片：
> 📝 您与 [张三] 的面试已于 15:00 结束，请及时评价：[晋级下一轮] [予以淘汰]

**第四步：多轮递进循环**
- 淘汰 → 触发标准淘汰闭环
- 晋级 → 自动归档当前轮次，新建下一轮记录（round_number +1），通知新面试官提供空闲时间
- 终审通过 → 进入 Offer 审批流程，AI 自动生成《候选人综合录用评估报告》，汇总所有轮次评价

### 3.3 四阶段看板

| Tab | 说明 | 展示内容 |
|-----|------|---------|
| 待预约 | 已通过筛选，等待选时间 | 可选时段列表 |
| 已预约 | 已选好面试时间 | 面试时间 + "等待面试" |
| 待评价 | 面试已完成，等待评价 | 面试时间 + 评价表单 |
| 已完成 | 已评价归档 | 通过/不通过 + 评价摘要 |

### 3.4 当前前端实现

- 面试官空闲时间录入（日期 + 时段勾选）
- 四 Tab 状态筛选：待预约 → 已预约 → 待评价 → 已完成
- 模拟面试完成 → 填写评价 → 通过/不通过

## 4. 数据大屏 — 全链路可视化

### 4.1 设计理念

大屏分为三大视觉区域：**核心 KPI（看总量）→ 流转漏斗（看状态）→ 时效警告（看异常）**。所有状态变更、时间戳最终汇聚到大屏，变成跳动的数字。

### 4.2 顶部：核心 KPI 计数看板

| 指标 | 说明 |
|------|------|
| 人才库总累积 | 底层数据库总记录数 |
| 当前活跃面试中 | 状态处于初试/复试/终审的总人数 |
| 今日安排面试 | 今天需要进行的面试场次 |
| 🚨 超时未处理 | 触发 7 天兜底机制、挂在共享池中的人数（红色高亮） |

### 4.3 中间层：状态流转与结构分析

- **招聘流转漏斗图**（倒金字塔）：按状态分组 —— 待筛选 → 初试中 → 复试中 → Offer → 已入职。哪个环节卡人最多一目了然。
- **在途状态环形图**（饼图）：当前活跃面试中的候选人状态占比，动态掌握面试工作负荷。
- **部门需求匹配分布**（横向柱状图）：按应聘部门统计在途人数，看出招聘热点与停滞部门。

### 4.4 底部：动态追踪与兜底红黑榜

- **今日面试动态滚动榜**：实时展示当日面试安排（时间、候选人、面试官、状态），方便前台接待掌握当天节奏。
- **🚨 7 天超时未响应红黑榜**：直接挂出超时名单（候选人姓名、滞留部门、滞留天数）。倒逼业务部门提升配合度。

### 4.5 当前前端实现

- **KPI 卡片**：待筛选 / 待预约 / 已通过 实时计数
- **ECharts 饼图**：候选人状态分布可视化，支持交互式图例切换
- 仅显示人数 > 0 的分类，自动过滤空数据

## 快速启动

```bash
# 安装依赖
npm install

# 启动开发服务
npm run dev
# → http://localhost:5173

# 构建生产版本
npm run build
```

项目目前为纯前端展示应用，无需后端服务。所有数据存储在浏览器 localStorage 中，首次启动自动加载 5 条示例候选人数据覆盖完整流程。

*本方案以"AI 大模型的聪明 + 传统后端逻辑的严谨"为设计哲学，既能以极低成本快速落地，又不牺牲数据确定性与流程闭环的完整性。*
